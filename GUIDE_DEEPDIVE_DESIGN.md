# 設計書: 文字起こしから guide.sh 水準の作業ガイドを作る仕組み

- 版: 1.0
- 作成日: 2026-07-23
- 前提: **情報源のベースは会議の文字起こし**（この前提だけは固定。それ以外は既存実装に縛られない）
- 到達目標: `/var/www/publish-cloudflare-workers-via-api/guide.sh` のように、**見るだけで「次に何をどうやるか」がパッとわかる**作業ガイドを、生成AIとの往復で作れる仕組み

---

## 0. 結論の要約

文字起こしから guide.sh 水準のガイドが一発で出ないのは、プロンプトが悪いからではない。
**guide.sh 水準に必要な情報（URL・画面表記・値の形式・検証方法・失敗時対処）の大半が、そもそも文字起こしの中に存在しない**からである。

したがって仕組みの中心は「もっと具体的に書き直して」という**書き直しループではなく、
足りない事実を集める取材（インタビュー）ループ**でなければならない。

この設計は次の3点を心臓部とする。

1. **知識台帳（Knowledge Ledger）** — ガイドに書く具体値をすべて「出典つきの事実」として1箇所に集め、
   confirmed / assumed / unknown の3状態で管理する。台帳にない具体値はガイドに書けない。
2. **取材ループ** — 生成AIの仕事を「文章を上手く書くこと」から
   「**最も価値の高い質問を、人が最も答えやすい形で作ること**」へ変える。
3. **準備度ゲート** — 「どのステップのどの情報が穴か」を機械的に採点・可視化し、
   埋まってから初めてガイド本文を生成する。ガイドの品質を運ではなく状態で管理する。

## 1. guide.sh の解剖 — 目標品質の正体

「パッとわかる」の正体を、guide.sh から性質として抽出する。これが品質基準の具体形であり、
後述のスロット（§4.2）と採点ルーブリック（§6）の根拠になる。

| # | 性質 | guide.sh での実例 |
|---|---|---|
| A | **全体地図が先にある** | 冒頭に Step 0〜6 の一覧と各ステップの完了/未完 |
| B | **各ステップ冒頭で「何が起き、何が変わらないか」を宣言** | 「このステップでは〜を確認します。Cloudflare のリソースや GitHub の設定はまだ変更しません」 |
| C | **ブラウザ操作は URL・選ぶ項目・入力値まで書く** | 「https://dash.cloudflare.com/profile/api-tokens 「Create Token」→「Create Custom Token」を選び…」 |
| D | **値は形式・例・取り違え注意まで書く** | 「32 文字の Account ID をコピーします。アカウント名や Zone ID ではありません」＋正規表現検証 |
| E | **実行できる操作は、内容を見せて確認を取ってから実行** | `run_inline "依存関係のインストール" npm ci` |
| F | **すべての手順に客観的な検証がある** | Token の有効性を API で確認、ポータルの応答文言を確認、cron 登録を確認 |
| G | **失敗時に何を確認しどこへ戻るかが書いてある** | 「dispatch_failed の場合は GH_DISPATCH_TOKEN の対象リポジトリ…を見直し、Step 2 と Step 5 を再実行します」 |
| H | **中断・再開できる** | `.setup-guide-state` に完了を記録し、再実行で未完了から再開 |
| I | **秘密情報・課金・不可逆操作の扱いを明示** | 「Token はファイルへ保存しません」「実行前に課金アカウントか再確認してください」 |

重要な観察: **C・D・F・G は「事実」がなければ絶対に書けない。** 文章力の問題ではない。
URL、画面の表記、値の形式、確認したときに返る値 — これらは調べるか、知っている人に聞くか、
実物を見るかでしか手に入らない。仕組みはこの「事実集め」を主役に据える。

## 2. なぜ文字起こしから一発で作れないか — 具体度の谷

文字起こしに含まれるのは **意図・合意・担当・期限・断片的な固有名詞** まで。
guide.sh 水準に必要な情報とのあいだには次の谷がある。

```
文字起こしにあるもの            ガイドに必要なもの
──────────────────            ──────────────────
「例の管理画面で設定変えて」 →  開くURL、ログインアカウント、選ぶメニューの正式表記
「あの値を直しておいて」     →  項目名、現在値、設定すべき値の形式と例、似た項目との区別
「終わったら共有して」       →  完了を客観的に判定する確認方法、共有先、共有形式
（言及なし）                 →  失敗時の切り戻し、必要権限、課金・不可逆操作の注意
```

生成AIがこの谷を越える方法は3つしかない。

- (a) **情報源から引く** — 正しいが、文字起こしだけでは足りない
- (b) **自分の世界知識から引く** — 一般的なサービスなら有効。ただし古い・環境違いの可能性があり、人の確認が必須
- (c) **創作する** — ハルシネーション

「もっと具体的に」と繰り返す書き直しループは、(a) が枯渇した時点で AI を (c) へ押し出す。
もっともらしい URL や画面名が混ざったガイドは、抽象的なガイドより危険である。

**設計原則: (a) を取材で拡張し、(b) は必ず人の確認を通し、(c) を構造的に禁止する。**

## 3. 仕組みの全体像

```
文字起こし（複数会議可）
   ↓
① 骨格抽出        … 作業の目的・完了状態・大まかなステップ列を抽出
   ↓
② 台帳初期化      … 文字起こし中の具体情報を「出典つき事実」として収穫し、
   │                 各ステップに必要なスロットの unknown を洗い出す
   ↓
③ 取材ループ ×N   … AIが質問バッチを作る → 人が答える/現物を貼る → 台帳更新 → 採点
   │                 （§5。この設計の中心。回数無制限、ただし収束制御あり）
   ↓
④ 準備度ゲート    … 全ステップのスロット充足を機械採点。穴はヒートマップで一目化。
   │                 通過条件: confirmed が閾値以上、または残りを人が明示的に
   │                 「作業前に確認」へ降格
   ↓
⑤ ガイド生成      … 台帳を唯一の事実源として直列化。台帳にない具体値の追加は禁止
   ↓
⑥ 机上実行レビュー … 別ロールのAIが「初見の作業者」としてステップを頭の中で実行し、
   │                 詰まる箇所を報告 → 未解決なら③へ還流
   ↓
⑦ 保存・人の承認  … 既存の needs_review → 承認フローに接続
   ↓
⑧ 実行フィードバック … 初回実行で「ガイド外参照」が発生した箇所を記録し、v2 の取材対象へ
```

ポイント: **③〜⑥は状態（台帳）を介したループ**であり、チャットの会話履歴には依存しない。
毎ラウンド、台帳全体をプロンプトに再投入するので、API自動でもコピー&ペーストでも同じに動き、
長い会話による文脈崩れ（context rot）も起きない。

## 4. 知識台帳（Knowledge Ledger）— 心臓部

### 4.1 エントリのスキーマ

```json
{
  "entryId": "L-012",
  "stepRef": "SK-3",
  "slot": "url",
  "claim": "APIトークン作成画面のURL",
  "value": "https://dash.cloudflare.com/profile/api-tokens",
  "evidence": {
    "type": "transcript | user_answer | artifact | ai_knowledge",
    "ref": "会議2026-07-20 14:32 の発言 / 回答A-005 / 貼り付け資料 / AI一般知識",
    "quote": "「トークンはプロフィールのとこから作るやつで」"
  },
  "confidence": "confirmed | assumed | unknown",
  "note": "取り違え注意: Global API Key の画面ではない"
}
```

### 4.2 スロット型 — 「guide.sh 水準」の機械的定義

§1 の性質 C・D・F・G・I をそのままスロットにする。**各ステップは以下のスロットが
埋まって初めて guide.sh 水準**であり、埋まり具合が準備度スコア（§6）になる。

| slot | 内容 | 由来 |
|---|---|---|
| `url` | 開く画面のURL（URLが存在しない作業は明示的に `該当なし` を confirmed にする） | 性質C |
| `ui_label` | 選ぶメニュー・ボタン・タブの画面表記どおりの名称 | 性質C |
| `input_value` | 入力・設定する値の形式・桁数・具体例・取り違え注意 | 性質D |
| `verification` | 完了を客観的に判定する方法（表示される文言 / 返る値 / 実行する確認操作） | 性質F |
| `failure_recovery` | 失敗時に確認する点と、どの手順からやり直すか | 性質G |
| `precondition` | そのステップに必要な権限・状態・事前準備 | 性質B/I |
| `caution` | 不可逆操作・課金・秘密情報・データ削除の注意 | 性質I |
| `decision_rule` | 判断が必要な箇所の選択肢と判断基準 | 品質基準 |
| `scope` | このステップで変わらないこと・やらないこと | 性質B |

### 4.3 3状態の運用ルール

| confidence | 意味 | ガイドへの反映 |
|---|---|---|
| `confirmed` | 出典があり、人が確認済み（本人回答・貼り付け現物・文字起こしの明確な発言） | 本文へそのまま書ける |
| `assumed` | AIの一般知識や推測による仮値。**必ず仮値だと明示する** | 確認されない限り本文に書けず、「作業前に確認: …（恐らく○○）」へ降格 |
| `unknown` | 値が無い | 質問になる。ゲート通過には解消か明示的降格が必要 |

**assumed を許すのがこの設計の要点。** 推測を全面禁止すると AI は質問を乱発して人が溺れる。
全面許可すると創作が混ざる。中間として「仮値は置いてよいが必ずタグを付け、確認質問に変換する」
ことで、AI の世界知識を安全に活用する（§2 の (b) ルート）。一般的な SaaS の設定画面 URL などは
仮値 → 人が「合ってる」と1クリックで confirmed になり、取材コストが激減する。

### 4.4 台帳の不変条件（アプリが機械検証する）

1. ガイド本文に現れる URL・画面名・具体値は、**対応する confirmed エントリが台帳に存在しなければならない**
   （直列化時にアプリが突合する。§7.3）。これがハルシネーションの構造的な防止策になる。
2. `unknown` と `assumed` は、ゲート通過時に必ず「解消」か「作業前に確認へ降格」のどちらかに落ちる。
   黙って消えることはない。
3. エントリの追加・確認・降格はすべて日時つきで記録し、既存の AI 対話履歴と同じ監査系に保存する。

## 5. 取材ループ — 生成AIとのやりとりの設計

ここが「生成AIとどう往復し、一緒に深掘りするか」の答えである。
1ラウンド = **AIの質問バッチ → 人の回答 → 台帳更新 → 採点** を、ゲート通過まで繰り返す。

### 5.1 1ラウンドの構造（AIの返答フォーマットは固定）

毎ラウンド、AI への入力は「品質基準＋骨格＋台帳全文＋前回の人の回答」。
AI の返答は次の6セクション固定とする（プロンプト全文は §10 P2）。

```
1. エコーバック   … 前回の回答をどう理解したかの要約（誤解をその場で発見させる）
2. 台帳更新       … 追加・確認・修正したエントリの差分（```json ブロック。アプリが取り込む）
3. 矛盾・気づき   … 台帳内の矛盾、文字起こしとの食い違い、リスクの指摘
4. 仮値の一覧     … いま assumed のまま残っている値の監査リスト
5. 準備度         … ステップ×スロットの充足サマリと前回からの増減
6. 次の質問       … 最大5問。§5.2 の質問設計原則に従う
```

フォーマットを固定する理由: (1) 台帳差分を機械で取り込める、(2) 人は毎回同じ場所を読めばよく
確認コストが下がる、(3) セクション4の仮値監査を毎回強制することで、推測の沈み込みを防ぐ。

### 5.2 質問設計の原則 — AIに埋め込む「取材術」

これをそのままプロンプトに書く（§10 P2 に反映済み）。

1. **想起させるな、確認させろ。**
   「設定画面のURLは何ですか？」ではなく
   「恐らく `https://dash.example.com/settings/api` だと思われます（一般的な構成からの推測）。
   合っていますか？ → はい / いいえ（正しいURLをそのまま貼ってください）」。
   人間は思い出す（recall）より見て判定する（recognition）方が速く正確。仮値はここで消化する。

2. **説明を求めるな、現物を求めろ。**
   「その画面はどんな画面ですか？」ではなく
   「その画面を開いて、**アドレスバーのURLをそのまま貼ってください**」
   「そのコマンドを一度実行して、**出力をそのまま貼ってください**」
   「その設定項目付近の**画面の文言をコピーして貼ってください**」。
   現物（アーティファクト）は嘘をつかず、画面表記どおりの `ui_label` が一度に手に入る。
   1つの貼り付けが複数スロットを同時に confirmed にする、最も効率の良い取材である。

3. **1ラウンド最大5問。影響度×不確実性の高い順。**
   人の作業記憶と根気には限りがある。質問には必ず「なぜ聞くか（どのステップのどの穴か）」を
   1行添える。答える気になる質問とは、意味がわかる質問である。

4. **選択式を優先し、自由記述は最後の手段。**
   はい/いいえ、3択、「次のうちどれですか」を基本形にする。

5. **ウォークスルーで暗黙知を掘る。**
   ステップ列そのものが怪しいときは、個別質問より先に
   「**この作業を頭の中で一度通しでやるつもりで、やることを順に書き出してください。
   雑でよく、思い出せない箇所は『？』と書いてください**」と依頼する。
   『？』の位置がそのまま次ラウンドの質問リストになる。認知タスク分析の定石であり、
   本人も気づいていない前提・順序依存が最もよく出る。

6. **矛盾は必ず質問化する。**
   台帳内・文字起こし間・回答間の食い違いは、黙って片方を採らず提示して裁定を仰ぐ。

7. **深さ優先への切り替え。**
   ステップ数が多いガイドでは、全体を薄く聞く（幅優先）ラウンドを1〜2回やった後、
   「今回はステップ3だけを完成させます」と宣言して1ステップ集中（深さ優先）に切り替える。
   散弾の質問より、人の頭の切り替えコストが小さい。

### 5.3 人の側の答え方（UIが誘導する）

- 各質問の下に「はい / いいえ / わからない / 作業前に確認でよい」ボタン＋自由記述欄。
- 「わからない」も立派な回答。AI は誰なら知っているか・どこを見れば分かるかを次ラウンドで提案する
  （例: 「この値は会議で山田さんが決めると言っています。確認して次回貼ってください」）。
- 「作業前に確認でよい」を選ぶと、そのスロットは正式に降格され、以後質問されない。
- 貼り付け現物は台帳に `artifact` 出典として保存し、参照資料スナップショットと同じ扱いにする。

### 5.4 収束の制御 — 往復を無限にしないために

- **効用の可視化**: 毎ラウンドの準備度スコア増分を表示。2ラウンド連続でほぼ増えなければ、
  アプリが「残りの unknown を作業前確認へ降格して先へ進みますか？」と提案する。
- **2回ルール**: 同じ質問が2ラウンド答えられなかったら、自動で降格候補にする。
  質問には限界効用があり、答えられない質問を持ち続けるとループ全体が腐る。
- **降格は敗北ではない**: guide.sh も「ブラウザで確認してください」を人に残している。
  重要なのは、**何が未確認かがガイドの前提条件に明示されている**ことであって、全知ではない。

## 6. 準備度ゲートと採点

### 6.1 採点（アプリが機械計算）

各ステップについて必須スロット（url / ui_label / input_value※入力がある場合 / verification /
precondition）＋リスクに応じた failure_recovery・caution の充足を数える。

```
準備度 = confirmed のスロット数 ÷ 必須スロット数   （ステップごと・全体）
```

### 6.2 ヒートマップ — 「どこが穴か」をパッとわかるように

ゲート画面はステップ×スロットのマトリクスで表示する。ガイドを読む前に、ガイドの穴が見える。

```
              url   ui   value  verify  fail   pre
ステップ1     ●     ●     ●      ●      ●      ●     100%
ステップ2     ●     ◐     ●      ○      ●      ●      67%   ← verify が unknown
ステップ3     ●     ●     ◐      ●      ○      ●      67%   ← value が assumed
   ● confirmed   ◐ assumed（要確認）   ○ unknown
```

### 6.3 通過条件

- 全ステップの必須スロットが「confirmed」または「人が明示的に降格した状態」になっている。
- 降格された項目は、生成されるガイドの **前提条件「作業前に確認: …」** に必ず現れる。

## 7. ガイド生成 — 台帳からの直列化

### 7.1 生成は「創作」ではなく「組版」

ゲート通過後の生成プロンプト（§10 P3）は台帳を唯一の事実源とし、
**台帳にない URL・画面名・具体値の追加を禁止**する。AI の仕事は事実の文章化と構成だけになる。
ここまで来れば生成は安定し、書き直しループは原理的に不要になる。

### 7.2 1ステップの解剖学（guide.sh 由来のテンプレート）

```
### Step N: <このステップの到達点を1行で>

このステップでは <何が起きるか>。<何は変わらないか（scope）>。

事前チェック:
  - <precondition。可能なものは実行/確認手順つき>

操作:
  1. <url> を開く
  2. 「<ui_label>」→「<ui_label>」を選ぶ
  3. <input_value: 形式・桁数・例・「〜ではない」の取り違え注意>

完了確認:
  - <verification: 表示される文言 / 返る値 / 実行する確認操作>

失敗した場合:
  - <failure_recovery: 確認する点と、どのステップからやり直すか>
```

### 7.3 直列化後の機械検証

既存のスキーマ検証・具体度チェックに加えて、**台帳突合**を行う:
ガイド本文中の URL と主要な具体値を抽出し、confirmed エントリに存在しないものがあれば
「出典不明の値」としてエラーにする。AI が最後の最後で創作を混ぜる事故を機械で止める。

### 7.4 スキーマ拡張（既存 JSON スキーマへの追加）

- `steps[].verification` : `{ "method": "visual | command | value_match", "detail": "…" }`
  — 既存 `completionCriteria` の構造化。command のものは将来 script 化候補（§9.2）。
- `steps[].failureRecovery` : `{ "checks": ["…"], "resumeFrom": "S2" }`
- `steps[].scopeNote` : このステップで変わらないこと
- `steps[].evidenceRefs` : 使用した台帳 entryId の配列（監査用）

## 8. 机上実行レビュー — 別ロールによる通し読み

生成直後、**生成とは別ロール・別コンテキスト**の AI に検収させる（§10 P4）。
同じ会話の中で「自分の書いたものを検収」させると、書いた経緯に引きずられて甘くなるため、
入力はガイド本文と品質基準だけにする（台帳もあえて渡さない — 初見の作業者と同じ条件にする）。

役割: 「このガイドだけを渡された初見の作業者」として頭の中で上から実行し、
**実行トレース形式**で詰まりを報告する。

```
Step 2 を実行中 → 「管理画面を開く」まではできた
→ 詰まり [MISSING_INFO]: どのアカウントでログインするか書かれていない
→ 詰まり [AMBIGUOUS_CHECK]: 「正しく設定されていることを確認」は何を見れば良いか判定できない
```

詰まりタイプ: `MISSING_INFO` / `AMBIGUOUS_CHECK` / `WRONG_ORDER` / `RISK_UNFLAGGED` / `JARGON`。
指摘は台帳の unknown として還流し、必要なら取材ループ（③）へ1〜2ラウンド戻る。
解消不能なら「作業前に確認」へ降格してから承認へ進む。

## 9. 実行フィードバック — ガイドは実行のたびに強くなる

### 9.1 ガイド外参照の記録

SPEC の成功基準1「ガイドのみで作業が完了する」をそのまま改善入力にする。
実行画面の各ステップに「**ガイド外の資料を見た / 人に聞いた**」ボタンを置き、
押されたら何を調べたかを1行だけ記録する。

実行完了後、この記録がそのまま **v2 の取材ループの初期質問リスト**になる。
「Step 3 で△△を調べた」→ その事実を台帳に confirmed で追加 → v2 生成。
2回目の実行では調べなくて済む。ガイドが実行のたびに guide.sh へ近づいていく。

### 9.2 script 化への接続

`verification.method = "command"` や、毎回同じ値を入れている input は、
既存の「スクリプト化候補の自動検出」の最有力候補としてそのまま挙がる。
guide.sh の「実行できるものは確認を取って実行する」（性質E）へ、登録スクリプトの範囲で近づく。

## 11. ミニ実例 — 1つの発言がガイドの1ステップになるまで

**文字起こし（断片）:**
> 「デプロイの前にステージングのフラグ切り替えるのやっといて。例の管理画面のやつ。」

**② 台帳初期化:**

| slot | value | confidence |
|---|---|---|
| url | （「例の管理画面」— 特定不能） | unknown |
| input_value | どのフラグをどの値へ？ | unknown |
| verification | ？ | unknown |

**③ 取材ラウンド1（AIの質問）:**

> （理由: ガイドの「開く画面」を特定するため）

> フラグ一覧画面の該当行の文言をコピーして貼っていただければ確実です。

**人の回答:** URL貼り付け＋「`enable_new_checkout` を ON にする」

**取材ラウンド2（AIの質問 — 仮値の確認と検証の取材）:**
> Q1 [SK-2 / verification] 切り替え後、反映を確認する方法は次のどれですか？
> (a) 同じ画面の表示が ON になればよい (b) ステージングサイトで新レジ画面が出ることを確認
> (c) 確認コマンドがある（あれば実行して出力を貼ってください）

**⑤ 生成されるステップ（guide.sh 水準）:**

```
### Step 2: ステージングの新チェックアウトを有効化する

  - https://admin.example.com/flags を開けること（社内SSOの権限が必要）

## 12. 既存実装との対応（参考 — 移行は段階的にできる)

既存の骨組みには残す価値のあるものが多い。置き換えるのは「深掘りの中身」である。

| 既存 | 扱い |
|---|---|
| 2形式保存・承認ゲート・監査履歴（AiInteractions）・スキーマ検証 | **そのまま残す** |
| プロバイダーアダプター / コピー&ペースト併存 | **そのまま残す**（本設計はラウンド単位で状態を持つため、どちらの経路でも同一に動く） |
| STEP 3–4（前提質問の一問一答） | **台帳＋取材ループ（§5）へ置き換え**。prerequisiteQuestions は初期 unknown として台帳へ取り込む |
| STEP 7 の設計案・深掘りプロンプト | **P2 ラウンドへ置き換え**。設計案の「## 7. まだ具体化できていない点」は台帳の unknown 一覧が代替する |
| `assessWorkGuideDepth_`（文字数・URL有無のヒューリスティック） | **残した上で、§6 のスロット採点と §7.3 の台帳突合を追加**。「短い説明」ではなく「出典のない値」を検出できるようになる |
| 全自動経路の maxAutoRefinements 往復 | **P4 机上実行レビュー→台帳還流に置き換え**。自動経路では人に聞けないため、unknown は最初から「作業前に確認」へ自動降格し、残存リスクに記録する |
| 実行画面 | **「ガイド外参照」ボタン（§9.1）を追加**し、v2 取材の入力にする |

データ面の変更は小さい: `WorkGuideBuildSessions.dataJson` に `skeleton` と `ledger` を持たせ、
ラウンドごとの P2 入出力を既存の AI 対話履歴（phase: `ledger_interview`)へ保存するだけでよい。

## 13. 設計原則の要約

```
ガイドの品質は文章力ではなく、集まった事実の量と確度で決まる
書き直しループではなく、取材ループを回す
AIの仕事は「上手く書くこと」より「良い質問を作ること」
具体値はすべて出典つきで台帳に置き、台帳にない値はガイドに書けない
推測は禁止せず、仮値として明示させ、確認質問で消化する
質問は 確認形式・現物要求・選択式・最大5問・理由つき
答えられない質問は「作業前に確認」へ降格する — 全知ではなく、未確認の明示が目標
生成の前にゲートを置き、穴はヒートマップで一目化する
検収は別ロール・初見の条件で、実行トレースとして書かせる
初回実行のガイド外参照を v2 の取材対象にし、ガイドを実行のたびに強くする
```
